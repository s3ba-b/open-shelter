# Glossary

Shared vocabulary for ShelterStack — domain terms a shelter worker would
use, and technical terms a developer needs. Kept short; if a term is widely
understood, it isn't listed.

## Domain terms

| Term | Meaning |
|---|---|
| **Shelter / rescue** | An animal welfare organization. In this system it is a **tenant** — its data is isolated from every other organization's. |
| **Tenant** | One isolated organization within the multi-tenant platform; the unit of data isolation. |
| **Intake** | The process and record of an animal entering a shelter's care. |
| **Adoption** | The transfer of an animal into a permanent home; tracked via an adoption application and approval flow. |
| **Adoption applicant** | A person applying to adopt; a data subject under [PRIVACY.md](PRIVACY.md). |
| **Foster / fostering** | Temporary care of an animal by a person (the **foster**) outside the shelter, recorded as a foster placement. |
| **Medical hold** | An animal status indicating it is not available pending medical treatment. |
| **Medical scheduling** | Vaccination and treatment schedules with due dates; reminders are generated by the background worker. |
| **Role** | A user's permission level within a single organization: **admin**, **staff**, or **volunteer**. |
| **Cross-tenant transfer** | A controlled, consent-gated move of an animal record (and its associated data) from one tenant to another (milestone M7). The only sanctioned crossing of the [tenant isolation](#technical-terms) boundary; see [PRIVACY.md](PRIVACY.md). |

## Technical terms

| Term | Meaning |
|---|---|
| **Multi-tenancy** | One running instance serving many isolated organizations (tenants). |
| **Tenant isolation** | The guarantee that one tenant can never read or modify another tenant's data — enforced at the data layer and verified by an automated cross-tenant test suite (a CI release gate). |
| **`ITenantContext`** | The resolved per-request tenant identity that scopes data access. |
| **Global query filter** | An EF Core filter applied automatically to every query for a tenant-scoped entity, so the tenant boundary can't be forgotten at a call site. |
| **.NET Aspire** | The orchestration framework that wires up the services and backing resources (PostgreSQL, Redis, broker) and provides the dashboard. |
| **AppHost** | The Aspire orchestration entry point (`src/ShelterStack.AppHost`) that runs the whole distributed app locally. |
| **ServiceDefaults** | Shared wiring (OpenTelemetry, health checks, resilience) used by every service (`src/ShelterStack.ServiceDefaults`). |
| **OpenTelemetry** | The instrumentation standard for traces, metrics, and logs, surfaced in the Aspire dashboard; spans here are tagged by tenant. |
| **Background worker** | A service (milestone M5) that runs scheduled jobs: reminders, follow-ups, per-tenant reporting, and housekeeping (e.g. retention). |
| **AGPL-3.0** | The project license; running it as a network service obliges the operator to offer the complete source to its users (§13). |
| **DCO** | Developer Certificate of Origin — sign off commits with `git commit -s` to contribute (see [CONTRIBUTING.md](CONTRIBUTING.md)). |

## Compliance terms

| Term | Meaning |
|---|---|
| **GDPR / RODO** | EU / Polish data-protection regulation. See [PRIVACY.md](PRIVACY.md). |
| **Controller** | The party deciding why and how personal data is processed — each **shelter**. |
| **Processor** | The party processing data on the controller's behalf — the **operator** of a shared hosted instance. |
| **Data subject** | A person whose personal data is processed (applicant, foster, volunteer, staff). |
| **DPA** | Data Processing Agreement (Art. 28), required between operator and each tenant. |
| **DPIA** | Data Protection Impact Assessment (Art. 35); see the decision record in [PRIVACY.md](PRIVACY.md). |
| **UODO** | The Polish data-protection supervisory authority. |
