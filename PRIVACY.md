# Privacy & GDPR (working reference)

> **Purpose.** This document captures Open Shelter's **privacy-by-design** approach
> so that privacy controls can be built into the product and handed to legal
> counsel for the eventual Privacy Policy, Terms of Service, and Cookie Policy. It
> expands the *Legal & compliance considerations* section of the
> [project charter](CHARTER.md).
>
> **This is not legal advice, and it is forward-looking.** A public launch (hosting
> for real organizations in the EU) is **out of scope** for the current version —
> see the charter's constraints. Policy wording, DPIA conclusions, and supervisory-
> authority questions must be confirmed by a qualified privacy lawyer before any
> public go-live in Poland / the EU.

## Roles & scope

In the multi-tenant model, a **tenant = a shelter / rescue organization**.

| Role | Who, in Open Shelter |
|---|---|
| **Controller** | Each **shelter / rescue** decides why and how its records are processed. |
| **Processor** | The **operator** that hosts a shared instance processes data on each shelter's behalf — which requires a **Data Processing Agreement (Art. 28)** with every tenant. |
| **Sub-processors** | Hosting, email, and other infrastructure providers — see [Sub-processors](#sub-processors--international-transfers). |
| **Data subjects** | People whose data the platform processes: adoption applicants, fosters, volunteers, and staff. |

**Self-hosting note:** an organization that self-hosts its own single-tenant
instance is its own controller *and* operator; the processor relationship above
applies to a shared, operator-hosted deployment.

## Personal data categories

Derived from the domain model (animals are not personal data; the **people**
attached to them are).

| Category | Examples | Special category (Art. 9)? |
|---|---|---|
| Account / staff | email, password hash, role (admin/staff/volunteer), name | No |
| Adoption applicant | name, contact details, application answers | No (re-assess if home-visit / health notes are added) |
| Foster | name, contact details, placement history | No |
| Volunteer | name, contact details | No |
| Attached documents | files attached to a record (e.g. an applicant's ID/proof document, a medical certificate/scan) — may contain any of the categories above | No (re-assess per attachment if health data about a person appears) |
| Technical | IP, user agent, auth logs | No |

## Purposes & legal bases (internal record — Art. 30 ROPA)

| Purpose | Data used | Legal basis (Art. 6) | Notes |
|---|---|---|---|
| Provide the service (manage animals, intake, adoptions, fostering, medical) | account + applicant/foster/volunteer data | (b) Contract / (f) Legitimate interest (per controller) | core function |
| Authentication & security | auth logs, IP | (f) Legitimate interest | minimal retention; see schedule |
| Operational notifications (e.g. medical-due, adoption follow-up) | contact details | (b) Contract / (f) Legitimate interest | generated by the M5 worker |
| Marketing (optional, future) | email | (a) Consent | double opt-in; out of MVP scope |

**Article 9:** Open Shelter does not intentionally process special-category data
today. Adding health/behavioral records about people (not animals) would trigger
a re-assessment and likely a DPIA — see [DPIA](#dpia-decision-record).

## Retention schedule

Retention automation is a natural job for the **M5 background worker** (housekeeping).

| Data | Retention | Automation (target) |
|---|---|---|
| Active account / records | while the tenant keeps them | — |
| Deleted record | erase / anonymize PII within a defined window | worker housekeeping job |
| Auth / security logs | e.g. 90 days | scheduled deletion |
| Application logs | e.g. 30–90 days; never passwords/tokens | log configuration |
| Password-reset / invite tokens | short-lived (e.g. 24h) | token expiry |
| Backups | rotation within a defined window | operational runbook |

## Data-subject rights → product feature → milestone

This is the most actionable table: it turns GDPR rights into engineering work.
In the MVP these operations are initiated by a **tenant admin via the staff web
app / API** (there is no public adopter portal in scope).

| Right (Art.) | Product feature | Milestone |
|---|---|---|
| Access / portability (15, 20) | export a specific person's data in a portable format | M4 (people records land with adoption/fostering) |
| Rectification (16) | edit an applicant/foster/volunteer/staff record | M4 |
| Erasure (17) | delete a person's data, with cascade / anonymization of related records | M4 + M5 (worker anonymization) |
| Restriction / objection (18, 21) | manual handling path; no marketing without consent | forward-looking |
| Complaint (77) | disclosed in the Privacy Policy (lead authority: **UODO**) | policy artifact |

**Attachments:** export and erasure cover any file attached to a person's record
(e.g. an uploaded ID/proof document) via the Attachment capability
([ROADMAP.md](ROADMAP.md) M2), not just structured fields.

**Audit trail:** every action in the table above (export, rectification, erasure) is recorded
in an immutable, per-tenant audit log (who, what, when) — see [ROADMAP.md](ROADMAP.md) M5.
This is a deliberate differentiator: incumbent shelter software (commercial or open-source)
does not expose this level of verifiable compliance.

**Response SLA:** one calendar month from a verified request (Art. 12(3)),
extendable for complex requests.

**Privacy contact:** `privacy@<domain>` — configure before any public launch.

## Security & breach handling

Extends the non-functional security requirements and the charter's
tenant-isolation work, which is itself an **Art. 32** compliance asset.

- HTTPS; password hashing; authorization; rate limiting.
- **Tenant isolation** at the data layer (EF Core global query filters over a
  resolved `ITenantContext`), with the cross-tenant isolation suite as a CI gate.
- Logs must never contain passwords, reset tokens, or full JWT/secret payloads.
- **Breach runbook (one page):** detect → contain → assess risk → if a risk to
  rights/freedoms is likely, notify the lead supervisory authority (**UODO**)
  within **72 hours** (Art. 33) → notify affected people without undue delay where
  risk is high (Art. 34) → record the incident (date, summary, actions).

(Chapter V).

## Minors

The staff-facing app is used by shelter personnel, not the general public, so
direct processing of minors' data is not expected. Confirm the minimum age and
any age-confirmation wording with counsel if that ever changes.

## EU public-launch checklist

- [ ] Privacy Policy + Terms of Service published, lawyer-reviewed
- [ ] DPA (Art. 28) offered to every tenant
- [ ] Record of processing (Art. 30) maintained
- [ ] Data export + account/record erasure implemented (see rights mapping)
- [ ] Retention documented; tokens/invites expire
- [ ] Logging minimized; breach runbook written
- [ ] Sub-processors listed; prod in EU/EEA or SCC in place
- [ ] No non-essential cookies without consent
- [ ] `privacy@` mailbox monitored
- [ ] No production PII in dev/test seed data
- [ ] AGPL §13 source offer actually provided to service users

## Open items for legal counsel

- Final Privacy Policy and Terms of Service wording
- Identity and address of the controller(s) / operator
- Lead supervisory authority and any DPO requirement
- Retention vs. anonymization decisions per record type
- Trigger for a full DPIA on scope expansion
- **Cross-tenant transfer (M7, post-v1):** a transfer moves a person's data from one
  **controller** (the source shelter) to another (the destination shelter) — this is a
  disclosure to a third-party controller, not just operator↔tenant processing, and needs its
  own legal-basis assessment and controller-to-controller terms before M7 is implemented.
